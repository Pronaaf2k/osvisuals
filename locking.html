Concurrency: Locking

Locks make shared data feel atomic.

Concurrent threads can interleave inside shared code. A lock protects the critical section so only one thread updates shared state at a time.

See Race Condition Compare Locks

Race Conditions

The bug is the interleaving.

Two threads both run cnt = cnt + 1. If each reads the old value before either writes, one increment disappears.

Implementation Path

From simple locks to fair locks.

Interrupt locks

Disable interrupts

Simple and useful on uniprocessors or MCUs, but not enough for multicore systems.

Faulty spin lock

Check then set

A thread can be preempted after the check, letting another thread also acquire.

Atomic spin lock

Test-and-set

Hardware atomically reads old value and writes held value in one indivisible step.

Queue lock

Park and unpark

Instead of wasting CPU by spinning, blocked threads sleep in a queue.

Atomic Primitive

Test-and-set makes acquire indivisible.

while (TestAndSet(lock, 1) == 1) {
  ; // spin
}

// only the winner enters
critical_section();
*lock = 0;

The thread that changes the lock from 0 to 1 owns the critical section. Others repeatedly observe 1 and wait.

CAS

Compare-and-swap is conditional.

Test-and-set blindly writes. Compare-and-swap writes only if the current value equals the expected value, making it more general for lock-free algorithms too.
