Persistence: File System

A file system turns raw blocks into named persistent files.

Storage hardware gives persistent blocks. The file system adds names, directories, metadata, inodes, allocation, file descriptors, links, and deletion semantics.

See Disk Layout Trace a Path

Three ways to identify a file

Name

Human path like /home/a.txt.

Inode + device

Persistent identity on disk.

File descriptor

Per-process handle with offset.

Opening a pathname does the expensive lookup once, then the process uses a small integer file descriptor for later read, write, seek, and close calls.

Inodes

An inode stores metadata, not the filename.

Metadata: size, owner, permissions, timestamps, link count.
Location: direct block pointers and indirect pointer blocks.
Directory entry: filename maps to inode number.

Multi-Level Index

Small files are cheap. Large files grow through indirection.

Direct pointers: first few data blocks are immediately reachable.
Single indirect: points to a block full of data-block pointers.
Double/triple indirect: expands capacity to gigabytes or terabytes with 4 KiB blocks.

Pathname Lookup

Opening a file walks directories until it finds the final inode.

A directory is itself a file whose data maps names to inode numbers.

Deletion and Links

There is no delete syscall, only unlink.

unlink: removes a directory entry and decrements inode reference count.
Hard link: another directory entry pointing to the same inode.
Soft link: a file containing another pathname; can become dangling.
Actual free: inode/data blocks are freed only when no directory refs and no open descriptors remain.
